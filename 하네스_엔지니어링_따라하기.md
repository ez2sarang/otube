## 1. 하네스 엔지니어링이란?

하네스 엔지니어링은 **AI에게 잘 일할 환경을 설계해주는 것**이다.

바쁜 팀장이 AI에게 일을 위임하고 싶다고 가정하자. 매번 "이번에는 이렇게 해줘", "저번에는 저렇게 해줘"라고 설명하는 것보다는 다음을 미리 정해주는 것이 훨씬 효과적이다:

- 우리 회사는 이렇게 일을 한다
- 우리 팀은 이런 방식으로 일한다
- Do and Don't가 있다
- 작업 지시서와 검증 기준이 있다

> **정의:** 하네스 엔지니어링은 LLM에게 업무를 위임하기 위하여 **규칙과 작업 구조, 검증 방법, 실행 흐름**까지 미리 묶어서 제어하는 것이다.

## 2. 프롬프트 엔지니어링 vs 하네스 엔지니어링

| 구분 | 프롬프트 엔지니어링 | 하네스 엔지니어링 |
|------|---------------------|-------------------|
| 방식 | 그때그때 질문을 던져서 결과를 얻음 | 환경 자체를 설계해서 일관된 결과를 얻음 |
| 특징 | AI를 도구처럼 사용 | AI를 팀원처럼 운용 |
| 한계 | 매번 맥락을 새로 설명해야 함 | 구조적으로 일관된 품질 유지 |
| 비유 | 그때그때 잘 요청하는 기술 | 일하는 환경 자체를 설계하는 작업 |

## 3. 하네스 엔지니어링의 4가지 구성 요소

### 3-1. 헌법 (Constitution)
- "우리 팀은 이렇게 일한다"라는 규칙을 적어놓은 문서
- LLM이 따라야 할 원칙과 규칙

### 3-2. 작업 구조 (Task Structure)
- 무엇을 어떻게 만들지 정의하는 작업 지시서
- 구체적인 목표와 범위를 명시

### 3-3. 검증 (Verification)
- 결과가 실제로 맞는지 체크하는 기준
- "이 결과물이 괜찮은지 아닌지 뭘 보고 판단할 건지"를 미리 정해놓는 것

### 3-4. 실행 루프 (Execution Loop)
- 수정 → 검증 → 반복의 흐름
- 고객/팀장/수요자인 내가 만족하는 결과물이 나올 때까지 반복

### 추가 요소

- **저장소 (Repository):** 작업을 어디서 할지. 로컬 폴더에서 시작해도 OK, Git으로 백업
- **참고 자료 (Reference):** 변수가 많고 복잡한 작업을 위한 레퍼런스

> **OpenAI 사례:** 빈 레포지토리에서 시작해 3명의 엔지니어가 5개월 동안 100만 줄의 코드를 작성했는데, 사람이 직접 쓴 코드는 단 한 줄도 없었다. 잘 짜인 하네스만으로 이 작업을 해냈다.

## 4. 레포지토리 하네스 vs 애플리케이션 하네스

### 레포지토리 하네스 = 회사의 취업 규칙
- 저장소 전체에 적용되는 공통 환경
- 문서 구조, 팀 규칙, 코드 제어(린터) 등

### 애플리케이션 하네스 = 팀 프로젝트 매뉴얼
- 프로젝트별로 만들고 싶은 앱/서비스에 대한 규칙
- 어떤 기능과 경험을 제공해야 하는지
- 해당 프로젝트에만 적용되는 규칙

### 접근 순서 (초보자용, 귀납적 방식)

개발자들은 레포지토리 하네스부터 짜지만, 처음 해보는 사람들은 "뭘 만들고 싶다"부터 시작하는 게 자연스럽다.

1. 만들고 싶은 앱을 먼저 상상
2. 애플리케이션 하네스를 만들며 구체화
3. 레포지토리 하네스로 확장

## 5. 실습: 사주 분석 앱 만들기

### 5-1. 준비물
- Claude Code 또는 Codex 같은 AI 코딩 도구
- 폴더 구조 템플릿

### 5-2. 아이디에이션 단계

1. OpenAI 팀이 만든 레포지토리 구조를 참고로 복사
2. Claude에게 붙여넣기 후 대화 시작:
   > "안녕 클로드야. 나는 사주분석 앱을 만들고 싶어. 직접 만드는 게 아니라 하네스 엔지니어링을 하고 싶어. 이 구조로 어떤 내용을 넣으면 좋을지 같이 의논해볼까?"
3. Claude가 선택지를 제공하며 질문 → 하나씩 답변하며 방향 설정
4. 하네스 초안을 함께 그려나감

### 5-3. 폴더 구조 (GitHub에 공개됨)

### 5-4. AGENTS.md 구조

```
Read Order:
1. harness/core/ → 저장소 공통 원칙 먼저 읽기
2. harness/workflows/ → 워크플로우 읽기
3. app/saju/harness/ → 앱별 규약 읽기
4. app/saju/plans/ → 현재 진행 트래커 읽기
```

### 5-5. 문서 검증 단계

문서를 다 만든 후 AI에게 검증 요청:

1. **논리 일관성 검토:** "여태까지 쓴 문서 다 검토해서 논리적으로 일관적인지, 모순이 없는지 재검토해줘"
2. **다관점 평가:** 시니어 개발자, 보안 담당자, 초보 개발자 관점에서 각각 평가
3. **점수 평가:** "N개의 기준을 제시하고 점수로 평가해줘" → 훨씬 정교한 평가 결과
4. **모호함 제거:** "500줄 이하로 짜라", "5번까지 시도 후 같은 문제가 남으면 기록해라" 등 명확한 기준 제공

### 5-6. 실행 및 결과

- Codex/Claude Code에서 폴더를 열고 실행
- AI가 문서를 읽고, 리뷰하고, MCP 서버 연결하고, 코드 작성
- Gradio 서버로 사주팔자 분석기 완성
- LLM API를 연결하면 더 복잡한 분석도 가능

### 5-7. Git 백업

- 여러 번 걸친 작업을 한 번에 복구 가능
- AI가 자율적으로 일할수록 Git 백업이 더 중요

### 5-8. 암묵지 좁히기

지식을 **암묵지**와 **명시지**로 나눌 수 있다:

- **명시지:** 글로 표현할 수 있는 지식
- **암묵지:** 머릿속에 있지만 AI에게 전달하지 않은 것

예시:
- "교육용이니까 당연히 코드가 짧고 읽기 쉬워야지" → AI는 모름
- "500줄 이하로 짜라"고 명시해야 AI가 따름
- 말을 안 해주면 AI는 "잘 돌아가는 코드"를 쓸 뿐, 10줄로 끝낼 수 있는 걸 300줄로 만들 수도 있음

## 6. 하네스 엔지니어링은 경영학이다

### 상사의 두 가지 타입

| 유형 | 특징 | 문제점 |
|------|------|--------|
| **방임형** | "AI는 유능하니까 알아서 잘하겠지" | 맥락 없이 일을 던짐, 원하는 결과 못 얻음 |
| **지시형** | 처음부터 모든 걸 다 설명 | AI가 더 나은 방법을 찾을 공간이 없음 |
| **위임형** (목표) | 목표는 명확하게, 방법론은 자율에 맡김 | 이것이 하네스 엔지니어링의 본질 |

### 위임이란?

- 수요자인 내가 결과물에 대한 명확한 상(像)을 가진 데서 출발
- 목표를 갖고 대리인에게 맡기되, 자율성과 방법론에 대한 권한을 보장
- 목표는 구체적이어야 하고, 명확하지 않으면 합의를 통해 맥락을 부여

### 피터 드러커의 MBO (Management by Objectives)

목표를 일방적으로 내려보내는 게 아니라 상사와 부하 직원이 함께 목표를 설정하고, 달성 여부로 성과를 평가하는 방식:

1. 목표 설정
2. 팀/개인 단위 쪼개기 (목표 캐스케이딩)
3. 개인 목표 합의
4. 실행 및 모니터링
5. 성과 평가

→ 이 과정이 하네스 엔지니어링과 똑같다.

### AI는 나를 조직으로 만든다

잭 샤피로(Jack Shapiro)의 "10x Lawyer" 글 참고:

- **기존 조직:** 암묵지가 사람 머릿속에 분산, 사람이 나가면 지식 소실
- **AI 시대:** 컨텍스트 윈도우가 A4 1.5장에서 소설책 6~7권 분량으로 성장 (2~3년 만에)
- **결과:** 한 사람이 팀의 아웃풋을 낼 수 있고, 개인이 기관 조직을 이길 수 있음

### 하네스 = 조직의 해자(Moat)

- 작업이 저장소에 축적되고
- 헌법과 작업 구조를 진화시킬 수 있고
- 검증 기준을 정교화할 수 있고
- 실행 루프가 점점 빠르게 돌아감
- **51번째 작업의 출발점은 50번째보다 훨씬 월등해진다**

> 경영학의 오랜 질문: "뛰어난 개인의 판단력과 암묵지를 어떻게 조직의 자산으로 만드는가?"
>
> 하네스 엔지니어링의 답: 헌법, 작업 구조, 검증, 실행 루프를 통해 코어 가치와 원칙을 외부화하고, 판단 패턴을 AI가 재사용 가능한 형태로 만들고, 검증 기준을 기계가 집행하게 하고, 매 작업마다 암묵지가 저장소에 축적되도록 한다.

## 7. 핵심 요약

1. **하네스 엔지니어링**은 AI가 잘 일할 수 있도록 환경을 설계하는 것
2. **4가지 구성 요소:** 헌법, 작업 구조, 검증, 실행 루프
3. 문서는 AI와 **합의하면서** 함께 만든다
4. 모호함을 없앨수록 AI가 더 잘 일한다
5. AI는 나를 **1인 조직**으로 만들 수 있다
6. 하네스는 그 조직의 **멘탈 모델**이다
7. 본질은 **위임형 상사가 되는 것**, 즉 나라는 조직을 경영하는 것
